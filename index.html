실시간 채팅 시스템 개발
STOMP WebSocket 기반 9종 이상 메시지 타입 지원. React Query 낙관적 업데이트로 서버 응답 전 UI 즉시 반영.
메시지 로딩 1초 이내 UI 즉각 반영 무한 스크롤
포인트
  • 클라이언트와 서버 상태 관리의 일관성 확보
  • 낙관적 업데이트 적용으로 사용자에게 즉각적인 UI 피드백 제공
기능 소개
  • STOMP WebSocket 기반 실시간 메시지 송수신 (텍스트, 이미지, 파일, 이모티콘, 멘션 등 9종 이상)
  • FlatList 무한 스크롤로 대용량 메시지 처리
  • React Query 캐시와 낙관적 업데이트로 UI 즉시 반영
  • 메시지 컴포넌트 구조의 확장성을 고려한 모듈화 설계
문제 해결
  • 기존 문제: 클라이언트 상태와 서버 상태가 분리되어 메시지 전송 후 서버 응답 대기 중 UI 반응 지연, 데이터 무결성 문제 발생
  • 개선: React Query 캐시를 단일 상태 소스로 활용. setQueryData를 통한 낙관적 업데이트로 메시지 전송 즉시 UI 반영
결과
  • 메시지 로딩 속도 1초 이내로 개선
  • UI 즉각 반영으로 사용자 경험 향상
  • 무한 스크롤로 메모리 효율적인 대용량 데이터 처리
성능 최적화
대용량 연락처 기반 초대 시스템
연락처 개수에 비례해 늘어나던 로딩 시간을 백그라운드 동기화 + 로컬 SQLite DB 캐싱으로 해결.
4,000건 25초 → 1초 이내 오프라인 지원 서버 부하 감소
개선 전 vs 후
  • 기존: 페이지 진입 시 대용량 연락처 API 호출 후 UI 렌더링 → 긴 로딩 시간, 서버 부하, UX 저하
  • 개선: 백그라운드(Headless JS)에서 100개 단위 청크 API 호출 → 로컬 SQLite DB 저장 → 페이지 진입 시 로컬 DB에서 즉시 렌더링
기술 스택
  • React Native, Native Modules (Android/iOS 연락처 접근 및 변경 감지), Headless JS, SQLite
  • 연락처 페이지 진입 시 로컬 DB 확인 후 ContactSyncTask 실행, 앱 포그라운드 복귀 시 자동 동기화
  • 네이티브 권한 체크, 네트워크 상태 확인, 실패 시 롤백 등 예외처리 적용
문제 해결 과정
  • 1단계: 전체 연락처 서버 전송 → 긴 로딩, 높은 서버 부하
  • 2단계: UI 선노출 + API 점진 호출 → 초기 UX 개선이나 API 요청량·검색 불일치 미해결
  • 3단계: 백그라운드 동기화 + 로컬 캐싱 도입 → UX 대폭 개선, 서버 부하 감소, 안정성 확보
결과
  • 연락처 4,000건 로딩: 25초 → 1초 이내로 획기적 개선
  • JS 스레드 블로킹 제거로 앱 성능 전반 향상
  • 오프라인 환경 지원, 서버 API 호출 수 대폭 감소 → 트래픽 및 비용 최적화

분산 알림 시스템 설계
포그라운드는 WebSocket + notifee, 백그라운드는 FCM으로 분리. TypeScript 타입 정의로 알림 구조 엄격 관리.

문제
  • 모든 알림을 FCM으로 통일 처리 → 포그라운드에서도 불필요하게 FCM 경유

해결 방안

  • TypeScript 구조화: NotificationClickEvent, NotificationListType 등 알림 소스·타입별 인터페이스 설계 → 컴파일 단계에서 오류 사전 차단
결과
  • 포그라운드 알림 속도 개선 및 FCM 리소스 절감
  • 다양한 알림 타입에도 일관된 구조 유지
  • TypeScript 기반 엄격한 타입 관리로 유지보수 부담 감소 및 확장성 확보

테스트를 작성한 실무적 이유
  • 인증 플로우는 모든 유저가 거치는 핵심 경로 - 깨지면 서비스 신뢰도에 직접 타격
  • 매 배포마다 동일 시나리오를 수동 반복하는 비생산적 작업을 자동화
  • Form validation·API 에러 처리를 시나리오로 고정 → "의도한 UX"를 테스트 코드로 문서화
실무에서 겪은 문제 & 해결
  • 플랫폼별 UI 동작 차이: iOS/Android 키보드 노출·스크롤 차이로 테스트 실패 → UI 상태 기반 테스트로 전환, "기기 의존적"에서 "상태 기반"으로 변경
  • flaky 테스트: API 응답 지연 + 애니메이션 타이밍 불일치 → 고정 delay 제거, API 응답 안정화 시점 기준으로 테스트 진행 → CI 환경에서도 안정적 통과
결과
  • 배포 전 인증 기능 회귀 여부를 수 분 내 확인 가능
  • 테스트 실패 지점이 UI 문제인지 API 문제인지 명확히 → 불필요한 커뮤니케이션 감소
  • QA 인력 없이도 핵심 유저 플로우에 대한 엔터프라이즈 수준의 안정성 확보

아키텍처
만보기 서비스 - SSOT 기반 아키텍처 재설계
Foreground Service 알림창 UI 추가로 발생한 데이터 정합성 문제. StepCounterManager 싱글톤 + useSyncExternalStore로 해결.
알림창/앱 UI 불일치 해결 위젯·워치 확장 가능
아키텍처 전체 설계
  • Android Native Module(AOSPedometer) 작성 - Foreground Service 걸음수 콜백 → RN 브리지 전달
  • Room DB · DAO · Repository 레이어 설계 및 StepCounterManager 코루틴 기반 데이터 동기화 구현
  • 재부팅 감지 BroadcastReceiver 방어코드로 서비스 재시작 시 걸음수 유실 방지
  • SSOT 아키텍처로 네이티브↔RN 간 걸음수 데이터 단일 출처 확보
문제의 발단
  • 신규 요구사항: 앱이 꺼져 있어도 Android 알림창(Foreground Service)에 걸음 수 실시간 표시
  • 발생 문제: 알림창 UI와 앱 화면이 각자 데이터를 처리 → 걸음 수 불일치(데이터 정합성 문제)
  • JS 레벨과 Native 레벨이 얽혀 기능 추가할 때마다 싱크 코드 중복, 관리 불가 상태에 도달
SSOT 아키텍처 설계
  • Native Layer: StepCounterManager 싱글톤 도입. Service·Module은 이 매니저를 '구독'만 하는 형태로 관심사 분리
  • JS Layer: 고빈도 데이터를 React 렌더링 엔진과 분리하기 위해 useSyncExternalStore 도입
  • 데이터 흐름 단방향화: Service → DB → React Native 방향으로 정리, 포그라운드 진입 시 DB 기준으로 UI 재동기화
결과
  • 알림창 / 앱 UI 간 걸음 수 불일치 완전 해결
  • 앱 재실행, 백그라운드 복귀 시에도 일관된 상태 유지
  • 위젯·워치 연동 요구사항이 와도 매니저만 연결하면 되는 확장성 확보
"기능의 확장은 아키텍처의 견고함을 시험하는 계기였습니다." - 네이티브와 JS 레이어 간의 명확한 역할 분담(Separation of Concerns)의 중요성을 깊이 깨달았습니다.
네이티브
네이티브 만보기 엔진 설계 - 삼성헬스·토스 벤치마킹
구글 헬스킷 의존성을 걷어내고 Hardware Sensor API를 직접 제어하는 자체 만보기 엔진 설계. OOM·재부팅 등 실환경 변수를 전방위 대응하여 24시간 끊김 없는 서비스 구현.
삼성헬스·토스 동일 정확도 24시간 서비스 안정성 재부팅 후 걸음수 유지
벤치마킹 및 목표
  • 삼성헬스·토스 앱 만보기 동작 방식 분석 → 하드웨어 센서 API 기반 자체 구현으로 동일 정확도 달성
  • 구글 헬스킷 외부 의존성 제거 → Native Sensor API 직접 제어로 플랫폼 종속성 해소
24시간 서비스 안정성
  • START_STICKY: 시스템이 메모리 부족으로 서비스를 종료하더라도 여유 시 자동 재시작 보장
  • OOM 방지: 만보기 포그라운드 서비스를 최우선 순위로 유지 - 리소스 부족 환경에서도 서비스 생존
  • OS 버전별 분기: Android 10(Q) ACTIVITY_RECOGNITION 권한 대응, Android 14/15 백그라운드 작업 제약 준수
걸음수 정확성 보장
  • 재부팅 대응: Native 센서 API는 재부팅 후 누적 걸음수를 리셋 → 자체 DB 로직으로 이전 값을 보존해 연속성 유지
  • 날짜 변경 감지: 자정 기준 걸음수 자동 초기화 처리
  • 백그라운드 재진입: 앱 재진입 시 서비스 누적값을 기준으로 JS 레이어 상태 보정
UX & 데이터 파이프라인
  • 실시간 UI 반영: DeviceEventEmitter로 서비스 → JS 즉시 전달, 앱 진입 시 걸음수 애니메이션 끊김 없음
  • 노티피케이션 설계: 포그라운드 서비스 유지 알림에 현재 걸음수 실시간 표시 - 토스 방식 채택으로 거부감 없는 UX 구현
  • DB 마이그레이션: 앱 업데이트 시 기존 수개월 치 걸음수 유실 방지 - SQLite 스키마 버전 관리 적용
결과
  • 삼성헬스·토스와 동일한 하드웨어 센서 API 기반으로 걸음수 정확도 일치 확인
  • 재부팅, OOM 등 실환경 변수 전방위 대응으로 24시간 서비스 안정성 확보
  • Native → JS 레이어 실시간 파이프라인으로 끊김 없는 걸음수 애니메이션 경험 제공

  • 이 서비스는 회원사마다 사용자 데이터·지원 기능이 달라, 크래시 발생 시 사용자 정보 + 회원사 정보를 함께 제출하는 구조로 설계
문제
  • Android 12+ 환경에서 ForegroundServiceStartNotAllowedException 등 비정상 종료·ANR 지속 발생
  • 네이티브 코드 수정 시 영향 범위 파악 어려움 → 크래시 발생 빈도 증가
대응 프로세스
  • Crashlytics를 통해 ANR / 크래시 탐지
  • 스택 트레이스 분석으로 문제 코드 식별
  • Android Studio Logcat / Profiler로 재현
  • Android OS 버전별 Foreground Service 실행 정책 분기 처리
  • 단계적 수정 후 재배포 및 재모니터링
결과
  • Foreground Service 관련 크래시 및 ANR 발생률 유의미하게 감소
  • Android 12+ 환경에서 서비스 안정성 확보
  • 회원사/사용자 정보 포함 크래시 리포트로 이슈 원인 특정 속도 대폭 향상

DevOps
배포 및 운영 자동화 - 빌드 파이프라인 현대화
EAS Build + OTA Updates + Fastlane으로 개발→문서→릴리즈 전 주기 자동화. 단순 수정 배포 시간 수일 → 수 분으로 단축.
수일 → 수 분 (OTA) 환경 차이 오류 0% 담당자 의존성 제거
문제 상황
  • 여러 협업 업체별로 환경이 달라 로컬 빌드 시 휴먼 에러 위험 높음
  • JS 코드 수정 한 줄에도 스토어 심사 전체 과정 반복 → 업데이트 주기 장기화
  • 스토어 콘솔 수동 업로드 및 메타데이터 수기 입력으로 개발자 공수 낭비
해결 방안
  • EAS 클라우드 빌드: eas.json으로 업체별/환경별 빌드 프로필 코드 관리 → 환경 불일치·휴먼 에러 원천 차단
  • OTA 업데이트 (EAS Updates): JS 레벨 수정은 스토어 심사 없이 즉시 반영. preview → release 단계적 배포로 리스크 최소화
  • Fastlane 자동화: EAS(빌드/업로드) + Fastlane(심사 제출·트랙 관리) 역할 분리 → 단일 명령으로 Android/iOS 심사 제출
  • Notion MCP 문서 자동화: 기능 개발 완료 시 API, 변경 사항이 Notion에 자동 기록되는 워크플로우 구축
결과
  • 단순 수정 배포 시간: 수일(심사 기간) → 수 분(OTA)으로 단축
  • 빌드 환경을 코드로 관리 → '환경 차이로 인한 크래시' 원천 차단
  • 릴리즈 담당자 개인 의존성 제거 → 배포 안정성과 운영 리스크 동시 감소
