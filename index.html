채팅
실시간 채팅 시스템 개발 (STOMP WebSocket & React Query)
React Query 캐시를 단일 상태 소스로 활용한 낙관적 업데이트로 메시지 UI 반영 속도 0ms 구현. Factory Pattern으로 9종 이상 복합 메시지 타입을 OCP 준수 구조로 확장.
UI 반영 0ms 메시지 로딩 1초 이내 9종+ 메시지 타입 무한 스크롤
기술 스택
  • React Native (FlatList), React Query, STOMP / WebSocket
① 낙관적 업데이트로 UI 반영 0ms 구현
  • 문제: 메시지 전송 후 서버 응답을 기다려야 UI가 갱신되어 사용자 경험 저하
  • 해결: React Query 캐시를 단일 상태 소스로 활용 - onMutate 단계에서 새 메시지를 캐시에 강제 삽입하여 즉각적인 피드백 제공
  • 롤백 처리: 전송 실패 시 onError 콜백에서 저장해둔 previousData로 캐시 복구 + '재전송' 버튼 노출로 데이터 유실 방지
② Factory Pattern 기반 메시지 컴포넌트 모듈화
  • 문제: 텍스트·이미지·파일·멘션 등 9종 이상 메시지 타입을 if-else 분기로 처리하면 타입 추가 시 기존 코드를 수정해야 하는 구조적 문제 발생
  • 해결: 메시지 타입(Key) → 렌더링 컴포넌트(Value)로 매핑하는 Message Renderer Factory 구축
  • 확장성: 새 메시지 타입 추가 시 맵에 컴포넌트만 등록하면 되는 OCP(Open-Closed Principle) 준수 설계 - 기존 로직 수정 불필요
결과
  • 메시지 송수신 UI 반영 속도 0ms 구현 (낙관적 업데이트)
  • 네트워크 불안정 환경에서도 롤백·재전송으로 안정적인 UI 유지
  • 메시지 로딩 체감 속도 1초 이내 개선
  • 9종 이상 복합 메시지 타입 대응을 위한 확장형 구조 완성
성능 최적화
비차단(Non-blocking) 아키텍처 - 연락처 기반 초대 시스템
메인 JS 스레드를 점유하던 4,000건 연락처 동기화를 Headless JS로 백그라운드 분리 + SQLite Offline-First 전략으로 로딩 시간 90% 이상 단축.
4,000건 20초 → 1초 이내 메인 스레드 점유 0% 오프라인 지원 서버 트래픽 최적화
문제 상황
  • 4,000건의 연락처 데이터를 매번 API로 가져오고 가공하는 과정에서 메인 JS 스레드 부하로 인한 프레임 드랍(Jank) 및 UX 저하 발생
  • 페이지 진입 시마다 대용량 API 호출 → 로딩 25초 이상, 높은 서버 부하
① Offline-First 전략 (SQLite)
  • API 호출 결과를 로컬 SQLite에 저장하여 Single Source of Truth로 활용
  • 앱 재진입 시 네트워크 없이도 로컬 DB에서 즉시 렌더링(First Paint)
  • Timestamp 비교를 통한 증분 업데이트(Incremental Update) - 변경된 데이터만 추적하여 불필요한 API 호출 제거
② UI 스레드 자유도 극대화 (Headless JS)
  • Headless JS를 활용하여 동기화 작업을 별도 백그라운드 프로세스로 분리 - 메인 스레드 점유율 0% 유지
  • 앱이 포그라운드에 없거나 사용자가 다른 화면을 보는 중에도 연락처 변경 사항 주기적 업데이트
③ Differential Sync (차분 동기화)
  • 핸드폰 연락처의 변경 사항(신규 추가 / 삭제) 감지
  • SQLite에 저장된 기존 데이터와 비교(Diffing)
  • 변경된 데이터(Delta)만 선택적으로 서버 API와 통신 → 서버 트래픽 및 배터리 소모 최적화
결과
  • 앱 재진입 시 데이터 로딩 시간 90% 이상 단축 (25초 → 1초 이내)
  • 메인 스레드 점유율 0%로 유지하며 백그라운드 동기화 성공
  • 오프라인 환경 지원 및 서버 API 호출 수 대폭 감소 → 트래픽·비용 최적화
아키텍처
SQLite 기반 Local-First 아키텍처 및 성능 최적화
서버 API 의존도를 최소화하고 로컬 SQLite를 단일 진실 공급원으로 활용. 오프라인 Action Queuing과 React Query 동기화로 네트워크 환경과 무관한 일관된 UX 제공.
초기 로딩 1.2s → 0.1s API 호출 70% 절감 오프라인 주요 기능 동작 UI 일관성 확보
기술 스택
  • SQLite (Local DB), React Query, Axios
① Single Source of Truth 설계 (서버 부하 절감)
  • 문제: 친구 목록·프로필·채팅방을 매번 API로 호출 → 서버 과부하 및 불필요한 트래픽 유발
  • 모든 데이터 흐름의 중심을 서버가 아닌 로컬 SQLite로 설정
  • 화면 진입 시 로컬 DB 즉시 조회하여 렌더링 → 백그라운드에서만 서버와 Revalidation 수행
  • 서버 API 호출 횟수 70% 이상 절감 → 인프라 비용 최적화
② 오프라인 모드 지원 및 UX 연속성 확보
  • Data Persistence: 채팅 메시지·사용자 설정을 SQLite에 영속화 → 오프라인 상태에서도 이전 데이터 조회 가능
  • Action Queuing: 오프라인 중 발생한 인터랙션(메시지 전송, 프로필 수정 등)을 로컬에 임시 저장 → 네트워크 복구 시 자동 동기화
  • 네트워크 단절이 에러 화면이 아닌 '지속 가능한 서비스 이용'으로 전환 → 이탈률 방지
③ 로컬 DB ↔ React Query 캐시 동기화
  • SQLite 업데이트 시 invalidateQueries 호출 또는 직접 캐시 갱신으로 UI와 DB 일관성 유지
  • 대량 채팅 데이터 조회 시 SQL INDEX 활용 + 페이징 처리를 SQL 레벨에서 수행 → 렌더링 성능 최적화
개선 결과 (Before → After)
  • 화면 초기 로딩: API 응답 대기 평균 1.2s → 로컬 조회 0.1s 이내
  • 오프라인 대응: '네트워크 연결 확인' 팝업 → 주요 기능(채팅, 목록 조회) 정상 동작
  • 서버 트래픽: 화면 진입 시마다 전체 호출 → 변경 데이터만 증분 업데이트(Delta)
  • 사용자 경험: 네트워크 환경 가변 → 어떤 환경에서도 균일한 성능 보장

하이브리드 알림 시스템: WebSocket & FCM 분산 설계
앱 상태(포그라운드/백그라운드)에 따라 알림 경로를 이원화. 포그라운드에서 FCM 중복 수신을 차단하고 로컬 알림으로 즉시 표시하여 지연 시간 0에 수렴.
포그라운드 알림 지연 0ms FCM 중복 호출 제거 서버 리소스 절감
문제 상황
  • 앱이 켜진 상태(소켓 연결 중)에서도 동일 내용의 FCM 푸시를 중복 수신 → 서버 리소스 낭비 및 미세한 알림 지연 발생
  • FCM은 구글 서버를 경유하므로 네트워크 상태에 따라 1~3초 지연 발생 가능
① 앱 상태별 알림 경로 이원화 (Hybrid Routing)
  • 포그라운드: WebSocket으로 수신한 데이터를 클라이언트에서 직접 가공 → Notifee로 로컬 알림 즉시 표시. 외부 푸시 서버 경유 없이 지연 시간 0에 수렴
  • 백그라운드: 소켓 연결이 끊긴 상태이므로 서버에서 FCM을 통해 푸시 전송 (기존 방식 유지)
② 전환 구간 알림 무결성 보장 (Handover Logic)
  • 포그라운드 ↔ 백그라운드 전환 순간 알림 유실·중복 위험 존재
  • 알림 고유 ID 기반 중복 여부 체크 로직 구현으로 데이터 무결성 보장
개선 결과 (Before → After)
  • 포그라운드 알림 속도: 평균 1~3초 (FCM 경유) → 즉시 발생 (로컬 트리거)
  • FCM 서버 부하: 모든 메시지당 1회 → 활성 사용자 수만큼 호출 감소
  • 데이터 수신: 소켓 + FCM 중복 수신 → 단일 경로 수신으로 최적화

인증/인가 E2E 테스트 자동화 (Detox)
1인 프론트엔드·QA 부재 환경에서 Detox로 인증 핵심 플로우를 자동화. 기획 변동과 서버 코드 변경 속에서도 배포를 두려움 없이 지속할 수 있는 지속 가능한 개발 환경 구축.
회귀 테스트 수분 내 자동화 배포 전 정합성 100% 포착 커뮤니케이션 비용 감소
① 의도한 UX의 명세화 및 문서화
  • 초기 프로젝트 특성상 복잡한 기획서 부재 → Form Validation·API 에러 처리 로직을 테스트 시나리오로 고정
  • 테스트 코드가 곧 최신화된 기능 명세서 역할 - 어떤 에러 상황에서 어떤 UI가 노출되어야 하는지 코드로 정의
② 클라이언트-서버 변경의 즉각적 트래킹
  • 프론트·백엔드 각 1명 환경 - API 스펙 변경이 클라이언트의 치명적 버그로 이어질 위험 상존
  • 서버 코드 변경 후 E2E 테스트 실행 → 배포 전 단계에서 정합성 문제 100% 포착
  • 테스트 실패 지점으로 UI 이슈인지 API 이슈인지 즉시 판별 → 백엔드 개발자와의 불필요한 디버깅 논의 시간 감소
③ 두려움 없는 기능 확장 (안전망 확보)
  • 기존 기능을 파괴하지 않고 새 기능을 추가할 수 있는 회귀 테스트 안전망 구축
  • "코드를 수정해도 기존 인증 플로우는 안전하다"는 확신 아래 리팩터링과 기능 고도화 가능
결과
  • 수동으로 확인하던 20여 개 인증 시나리오 → 수 분 내 자동 검증으로 전환
  • 배포 전 클라이언트-서버 정합성 문제 100% 사전 포착
  • QA 인력 없이도 핵심 플로우에 대한 엔터프라이즈 수준의 안정성 확보

  • 포인트 적립이라는 확실한 보상을 통해 광고에 대한 유저 거부감을 최소화

  • "포인트 = 현금" 인식을 심어 포인트 실사용 전환율과 앱 의존도 상승

아키텍처 전체 설계
  • Android Native Module(AOSPedometer) 작성 - Foreground Service 걸음수 콜백 → RN 브리지 전달
  • Room DB · DAO · Repository 레이어 설계 및 StepCounterManager 코루틴 기반 데이터 동기화 구현

문제의 발단
  • 신규 요구사항: 앱이 꺼져 있어도 Android 알림창(Foreground Service)에 걸음 수 실시간 표시
  • 발생 문제: 알림창 UI와 앱 화면이 각자 데이터를 처리 → 걸음 수 불일치(데이터 정합성 문제)
  • JS 레벨과 Native 레벨이 얽혀 기능 추가할 때마다 싱크 코드 중복, 관리 불가 상태에 도달
SSOT 아키텍처 설계
  • Native Layer: StepCounterManager 싱글톤 도입. Service·Module은 이 매니저를 '구독'만 하는 형태로 관심사 분리
  • JS Layer: 고빈도 데이터를 React 렌더링 엔진과 분리하기 위해 useSyncExternalStore 도입
  • 데이터 흐름 단방향화: Service → DB → React Native 방향으로 정리, 포그라운드 진입 시 DB 기준으로 UI 재동기화
결과
  • 알림창 / 앱 UI 간 걸음 수 불일치 완전 해결
  • 앱 재실행, 백그라운드 복귀 시에도 일관된 상태 유지
  • 위젯·워치 연동 요구사항이 와도 매니저만 연결하면 되는 확장성 확보
