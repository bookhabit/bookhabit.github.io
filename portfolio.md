## 프로젝트 경험 (비즈비톡)

### 실시간 채팅 시스템 개발

**포인트**

- 클라이언트와 서버 상태 관리의 일관성 확보
- 낙관적 업데이트 적용으로 사용자에게 즉각적 UI 피드백 제공

**기능 소개**

- STOMP WebSocket 기반 실시간 메시지 송수신 (텍스트, 이미지, 파일, 이모티콘, 멘션 등 9종 이상 메시지 타입 지원)
- FlatList 무한 스크롤로 대용량 메시지 처리
- React Query 캐시와 낙관적 업데이트로 UI 즉시 반영
- 메시지 컴포넌트 구조 확장성 고려한 모듈화 설계

**문제 해결**

- **기존 방식 문제**
  1. 클라이언트 상태와 서버 상태가 분리되어 있어 메시지 전송 후 서버 응답 대기 중 UI 반응 지연
  2. 데이터 무결성 문제
- **개선 방식**
  1. React Query 캐시를 단일 상태 소스로 활용
  2. `setQueryData`를 통한 낙관적 업데이트 적용으로 메시지 전송 즉시 UI 반영

**결과**

- 메시지 로딩 속도 1초 이내로 대폭 개선
- UI 즉각 반영으로 사용자 경험 향상
- 무한 스크롤로 메모리 효율적 대용량 데이터 처리 가능

### 대용량 연락처 기반 초대 시스템

**포인트**
연락처 개수에 비례해 선형적으로 늘어나던 로딩 시간 문제를 백그라운드 동기화 및 로컬 DB 캐싱으로 해결

**기능 소개**

- **기존**: 페이지 진입 시 대용량 연락처 API 호출 후 UI 렌더링 → 긴 로딩시간, 서버 부하, UX 저하
- **개선**: 백그라운드(Headless JS)에서 100개 단위 청크 API 호출하여 로컬 SQLite DB에 저장 → 페이지 진입 시 로컬 DB에서 즉시 UI 렌더링, 서버 호출 최소화

**기술 스택 및 업무**

- React Native, Native Modules (Android/iOS 연락처 접근 및 변경 감지), Headless JS, SQLite, MMKV, Chunked API 요청(100개 단위)
- 연락처 페이지 진입 시 로컬 DB 데이터 확인 후 ContactSyncTask 실행, 앱 포그라운드 복귀 시 동기화 실행, 네이티브 모듈로 연락처 변경 감지 및 동기화 트리거
- 네이티브 권한 체크, 네트워크 상태 확인, 실패 시 롤백 등 예외처리 적용

**문제 해결 과정**

1. 전체 연락처 서버 전송 방식 → 긴 로딩, 높은 서버 부하 문제 발생
2. UI 선노출 + API 점진 호출 방식 → 초기 UX 개선이나 API 요청량, 검색 불일치 문제 미해결
3. 최종 백그라운드 동기화 + 로컬 캐싱 도입 → UX 대폭 개선, 서버 부하 감소, 안정성 확보

**결과**

- 연락처 4,000건 로딩 속도 25초 → 1초 이내로 획기적 개선
- JS 스레드 블로킹 제거로 앱 성능 전반 향상
- 백그라운드 동기화로 앱 진입 시 즉시 UI 제공, 오프라인 환경 지원
- 연락처 변경 감지 → 디바이스 연락처와 서버 DB 동기화 → 서버 API 호출 수 대폭 감소, 트래픽 및 비용 최적화

**얻은 경험**

- 대용량 데이터 백그라운드 처리 및 동기화 전략 실전 적용 경험
- Native 모듈과 React Native 연동, Headless JS 활용 능력 강화
- 사용자 경험 중심 대용량 데이터 처리 설계 및 서버 부하 최적화 경험

### Offline 지원 및 성능개선을 위한 SQLite 도입

**포인트**
React Native 앱에 SQLite 기반 로컬 DB를 도입하여 서버 API 의존도를 줄이고, 오프라인 환경에서도 빠르고 안정적인 UX를 제공하는 로컬 우선 구조 구현

**기능 소개**

- **기존 방식**: 화면 진입 시마다 서버 API 호출 → 로딩 지연 및 불필요한 네트워크 요청 다수 발생
- **개선 방식**: 로컬 DB에 데이터 저장 후, UI는 로컬 DB 우선 조회 → 서버 호출은 비동기 처리로 UX 지연 최소화

**결과**

- 오프라인 상태에서도 주요 기능 정상 동작 보장
- 서버 API 호출 횟수 및 트래픽 대폭 감소
- 실시간 데이터 변경 시 로컬 DB 및 React Query 캐시 동기화로 UI 일관성 확보

### 알림 기능

**알림 시스템 구조 설계 및 개선**

**포인트**

- FCM 리소스 낭비를 줄이기 위해 **분산 알림 구조** 설계
  - 포그라운드에서 소켓으로 받은 메시지는 로컬 알림(notifee)으로 처리
  - 백그라운드 알림은 FCM으로 처리
- 메시지 외 알림 타입이 다양해져 복잡해진 구조를 **TypeScript 타입 정의**로 엄격히 관리하여 컴파일 단계에서 오류를 사전에 차단, 유지보수성 향상

**문제 및 해결방안**

1. **문제**
   - 모든 알림을 FCM으로 통일해 처리 → 포그라운드에서도 불필요하게 FCM 경유
   - 리소스 낭비 및 반응 지연 발생
   - 알림 설정 반영 로직이 단순해 유지보수는 쉽지만 성능 효율이 떨어짐

2. **해결방안**
   - **분산 처리 방식** 채택
     - 포그라운드: WebSocket으로 메시지 수신 후 로컬 알림(notifee) 즉시 표시
     - 백그라운드: FCM 푸시 알림 수신 후 표시
   - 알림 소스·타입별로 구조화된 TypeScript 인터페이스 설계
     - `NotificationClickEvent`, `NotificationListType` 등으로 알림 데이터 형식 표준화 및 확장성 고려

**결과**

- 포그라운드 알림 속도 개선 및 FCM 리소스 절감
- 다양한 알림 타입에도 일관된 구조 유지 가능
- TypeScript 기반의 엄격한 타입 관리로 유지보수 부담 감소 및 확장성 확보

### 인증/인가 E2E 테스트

#### 1. 테스트 코드를 작성한 실무적 이유

##### ① QA 리소스를 "사람"이 아닌 "시스템"으로 대체

인증(회원가입 / 로그인)은

- 모든 유저가 반드시 거치는 핵심 플로우
- 한 번 깨지면 서비스 신뢰도에 직접적인 타격이 발생

하지만 실무에서는

- QA 인력이 충분하지 않거나
- 개발자가 직접 수동 테스트를 반복하는 경우가 많음

이로 인해 매 배포마다:

- 동일한 시나리오를 직접 입력
- 버튼 클릭
- 에러 케이스 재확인

하는 **비생산적인 작업**이 반복됨

→ **E2E 테스트로 인증 플로우를 자동화하여, QA 시간을 코드 실행 시간으로 치환**

##### ② "리팩터링이 두려운 코드"를 "변경 가능한 코드"로 만들기

인증 로직은:

- UI 변경
- API 스펙 변경
- 에러 처리 로직 추가

등의 수정이 빈번함

하지만 테스트가 없을 경우:

- 작은 UI 수정에도
- "로그인이 혹시 깨지지 않았을까?"라는 불안으로

수동 검증이 필수

→ E2E 테스트를 통해 **리팩터링 이후에도 핵심 기능이 정상 동작함을 즉시 검증 가능**

##### ③ Form / API 에러 처리를 명세 수준으로 고정

실무에서 자주 발생하는 문제:

- Form validation 규칙이 코드마다 다르게 적용됨
- API 에러 메시지 처리 방식이 화면마다 달라짐

E2E 테스트에서:

- 필수 입력값 누락
- 잘못된 형식 입력
- API 실패 응답

을 **시나리오로 고정**

→ "의도한 UX"가 테스트 코드로 명확히 문서화됨

#### 2. E2E 테스트에서 실제로 겪은 문제와 대응 (실무 디테일)

##### 문제 ① 플랫폼별 UI 동작 차이로 테스트 실패

- iOS / Android 간
  - 키보드 노출 방식
  - 입력 포커스 이동
  - 스크롤 위치

차이로 동일한 테스트 스크립트가 실패

**대응**

- 입력 후 스크롤 이동 로직 명시
- UI 상태를 기준으로 다음 액션 수행하도록 수정

→ 테스트가 "기기 의존적"이 아닌 "상태 기반"으로 동작

##### 문제 ② API 응답 지연 + 애니메이션 타이밍으로 flaky 테스트 발생

- 네트워크 상태에 따라
  - API 응답 타이밍 변동
  - 애니메이션 완료 전 assertion 실행

**대응**

- 고정 delay 제거
- API 응답 안정화 시점 기준으로 테스트 진행
- 이메일 인증 플로우는 "발송 성공 → 인증 완료 상태 가정"으로 분리 테스트

→ flaky 테스트 제거, CI 환경에서도 안정적으로 통과

#### 3. 실질적인 성과 (가장 중요한 부분)

##### ① 배포 전 인증 기능 회귀 여부를 **수 분 내 확인 가능**

- 리팩터링
- UI 수정
- API 변경

이 발생해도

→ **테스트 실행만으로 로그인/회원가입 정상 여부 즉시 판단**

##### ② API 변경 시 프론트/서버 커뮤니케이션 비용 감소

- 테스트 실패 지점이 곧
  - UI 문제인지
  - API 응답 문제인지

를 명확히 알려줌

→ "이거 서버 문제인가요?" 같은 불필요한 커뮤니케이션 감소

##### ③ QA 인력이 없어도 신뢰 가능한 검증 체계 확보

- 사람이 직접 테스트하지 않아도
- 핵심 유저 플로우에 대해
  - 일관된 품질 보장 가능

→ 소규모 팀 / 1인 개발 환경에서도 **엔터프라이즈 수준의 안정성 확보**

#### 4. 한 줄 요약 (포트폴리오 / 면접용)

**포트폴리오용**

> 인증 기능을 E2E 테스트로 자동화하여, QA 리소스 없이도 핵심 유저 플로우의 안정성을 지속적으로 검증할 수 있는 구조를 구축했다.

**면접 구두 설명용**

> 테스트 코드를 작성한 이유는 '버그를 찾기 위해서'가 아니라, **리팩터링과 배포를 두려움 없이 할 수 있는 환경을 만들기 위해서**였습니다.

## 프로젝트 경험 (복지찬스)

### 만보기 서비스 운영 — SSOT 기반 아키텍처 설계 및 안정화

**서비스 개요**

- React Native + 네이티브(Android/iOS) 하이브리드 기반 만보기 서비스
- Android Foreground Service를 활용한 백그라운드 걸음 수 측정
- 실제 사용자 대상 운영 경험 보유 (3개월+)

**문제의 발단**

- **기존 구조**: 앱이 켜져 있을 때만 JS 레이어에서 걸음 수를 계산하여 화면에 표시
- **신규 요구사항**: "앱이 꺼져 있어도 사용자가 실시간으로 걸음 수를 확인할 수 있도록 **Android 알림창(Foreground Service)에 UI를 노출**하라"
- **발생한 기술적 난관**: 알림창 UI와 앱 내부 화면이 각자 데이터를 처리하면서 걸음 수가 일치하지 않는 **데이터 정합성(Data Integrity)** 문제 발생

**깨달음: 관심사 분리의 부재가 유지보수를 어렵게 만든다**

- JS 레벨과 Native 레벨이 복잡하게 얽혀 있어, 기능 하나를 추가할 때마다 양쪽의 싱크를 맞추는 코드가 중복되고 관리가 불가능해짐을 체감
- "변경에 유연한 구조 없이는 OS의 생명주기(Foreground/Background) 변화를 감당할 수 없다"는 결론 도출

**아키텍처 혁신: SSOT(Single Source of Truth) 설계**

- **Native Layer**: 데이터의 '생성'과 '보존'을 전담하는 `StepCounterManager` 싱글톤 도입. Service와 Module은 이 매니저를 '구독'만 하는 형태로 구조 개편 (관심사 분리)
- **JS Layer**: 고빈도 데이터를 React 렌더링 엔진과 분리하기 위해 `useSyncExternalStore` 도입
- **데이터 흐름 단방향화**: Service → DB → React Native 흐름으로 정리, 앱 포그라운드 진입 시 DB 기준으로 UI 상태 재동기화

**결과**

- 알림 센터 / 앱 UI 간 걸음 수 불일치 문제 해결
- 앱 재실행, 백그라운드 복귀 시에도 일관된 상태 유지
- 향후 위젯(Widget)이나 워치(Watch) 연동 요구사항이 와도 매니저만 연결하면 되는 **확장성** 확보

> **"기능의 확장은 아키텍처의 견고함을 시험하는 계기였습니다."**
> 단순한 화면 표시를 넘어 '상주형 알림 서비스'로 기능을 확장하며 발생한 데이터 불일치 문제를 통해, 네이티브와 JS 레이어 간의 명확한 역할 분담(Separation of Concerns)의 중요성을 깊이 깨달았습니다.

### 네이티브 만보기 엔진 설계 — 삼성헬스·토스 벤치마킹

**목표 및 접근**

- 삼성헬스·토스 앱 만보기 동작 방식 분석 → 하드웨어 센서 API 기반 자체 구현으로 동일 정확도 달성
- 구글 헬스킷 외부 의존성 제거 → Native Sensor API 직접 제어로 플랫폼 종속성 해소

**24시간 서비스 안정성**

- **START_STICKY**: 시스템이 메모리 부족으로 서비스를 종료하더라도 여유 시 자동 재시작 보장
- **OOM 방지**: 만보기 포그라운드 서비스를 최우선 순위로 유지 — 리소스 부족 환경에서도 서비스 생존

- **OS 버전별 분기**: Android 10(Q) `ACTIVITY_RECOGNITION` 권한 대응, Android 14/15 백그라운드 작업 제약 준수

**걸음수 정확성 보장**

- **재부팅 대응**: Native 센서 API는 재부팅 후 누적 걸음수를 리셋 → 자체 DB 로직으로 이전 값을 보존해 연속성 유지
- **날짜 변경 감지**: 자정 기준 걸음수 자동 초기화 처리
- **백그라운드 재진입**: 앱 재진입 시 서비스 누적값 기준으로 JS 레이어 상태 보정

**UX & 데이터 파이프라인**

- **실시간 UI 반영**: `DeviceEventEmitter`로 서비스 → JS 즉시 전달, 앱 진입 시 걸음수 애니메이션 끊김 없음
- **노티피케이션 설계**: 포그라운드 서비스 유지 알림에 현재 걸음수 실시간 표시 — 토스 방식 채택으로 거부감 없는 UX 구현
- **DB 마이그레이션**: 앱 업데이트 시 기존 수개월 치 걸음수 유실 방지 — SQLite 스키마 버전 관리 적용

**결과**

- 삼성헬스·토스와 동일한 하드웨어 센서 API 기반으로 걸음수 정확도 일치 확인
- 재부팅, OOM 등 실환경 변수 전방위 대응으로 24시간 서비스 안정성 확보
- Native → JS 레이어 실시간 파이프라인으로 끊김 없는 걸음수 애니메이션 경험 제공

### Firebase Crashlytics 기반 ANR 모니터링 체계 구축

**서비스 규모**

- 실사용자 1만 명 규모의 프로덕션 서비스를 6개월간 유지보수 및 안정화

**핵심 인사이트**

- **Release 빌드에서만 재현되는 크래시는 Crashlytics 없이 원인 특정 불가**하다는 경험이 가장 핵심
- 크리티컬한 실수 또는 예상치 못한 예외 상황이나 기능 추가로 인해 구조가 변경될 수 있다는 점 → 모니터링 환경 구축이 필수

**문제 정의**

- Android 12+ 환경에서 Foreground Service 실행 중 비정상 종료 및 ANR 지속 발생
- 대표 오류: `ForegroundServiceStartNotAllowedException`
- 네이티브 코드 수정 시 영향 범위 파악 어려움 → 크래시 발생 빈도 증가

**모니터링 환경 구축 전략**

- Firebase Crashlytics 연동: 크래시, ANR, 네이티브 스택 트레이스 수집
- **이 서비스는 회원사마다 사용자 데이터와 지원 기능이 다르기 때문에**, 크래시 발생 시 어떤 사용자인지 한눈에 알아볼 수 있도록 **사용자 정보 + 회원사 정보를 함께 제출**하는 구조 설계
- Android Vitals 지표와 Crashlytics를 병행하여 이중 모니터링

**대응 프로세스**

1. Crashlytics를 통해 ANR / 크래시 탐지
2. 스택 트레이스 분석으로 문제 코드 식별
3. Android Studio Logcat / Profiler로 재현
4. Android OS 버전별 서비스 실행 정책 분기 처리
5. 단계적 수정 후 재배포 및 재모니터링

**결과**

- Foreground Service 관련 크래시 및 ANR 발생률 유의미하게 감소
- Android 12+ 환경에서 서비스 안정성 확보
- 운영 중 장애를 모니터링 기반으로 빠르게 대응 가능한 구조 구축
- 회원사/사용자 정보 포함 크래시 리포트로 이슈 원인 특정 속도 대폭 향상

### 구글 최신 정책 대응 — Expo SDK 마이그레이션 (v51 → v53) 및 New Architecture 전환

**배경**

- Google Play 스토어 정책: Android 15(API 35) 타겟팅 의무화
- Expo SDK 53 기본 설정: **React Native New Architecture** (Fabric + TurboModules) 활성화
- 향후 도입될 **16KB 페이지 크기 환경** 대응 필요

**수행 내용**

- **Android 15 대응**: 타겟 SDK를 35로 상향 조정, 관련 권한 및 백그라운드 작업 최적화
- **New Architecture 전환**: 기존 Bridge 방식에서 JSI(JavaScript Interface) 기반의 동기식 네이티브 호출 구조로 전환
  - 비동기 브릿지 병목 현상 해결 → 앱 응답 속도 개선
  - Fabric Renderer 전환으로 UI 렌더링 성능 최적화
- **16KB 페이지 지원**: 네이티브 종속성 전수 조사 및 메모리 정렬 이슈 해결
- **마이그레이션 관리**: `expo-upgrade-helper` 기반으로 체계적 업데이트 진행, 뉴 아키텍처에서 지원이 변경된 서드파티 라이브러리 선별 후 대체 또는 리팩토링

**결과**

- Google Play 배포 규정 준수 및 최신 OS 호환성 확보
- JSI 기반 네이티브 호출로 앱 응답 속도 개선
- 체계적인 업데이트 프로세스로 배포 안정성 확보
- 메이저 업데이트 후 발생하는 라이브러리 파손(Breaking Changes)을 즉각 수정하여 사용자 이탈 없는 서비스 운영

### 배포 및 운영 자동화 — 빌드 파이프라인 현대화

> 배포(Fastlane)와 개발 문서(Notion MCP)를 각각 자동화한 것이 아니라, **개발 → 문서 → 릴리즈까지 사람이 개입하지 않아도 일관되게 동작하는 운영 체계를 구축**했다.

**문제 상황**

- 여러 협업 업체별로 환경이 달라 로컬 빌드 시 환경 설정 실수(휴먼 에러) 위험 높음
- JS 코드 한 줄 수정에도 스토어 전체 심사 및 배포 과정 반복 → 업데이트 주기 장기화
- 스토어 콘솔 수동 업로드 및 메타데이터 수기 입력으로 인한 개발자 공수 낭비

**해결 ① — EAS 클라우드 빌드 환경 구축**

- Expo EAS Build 기반 클라우드 빌드 환경 도입
- `eas.json`으로 업체별/환경별(Development, Preview, Production) 빌드 프로필을 코드로 관리 → **환경 불일치 및 휴먼 에러 원천 차단**
- 플랫폼별 빌드 프로파일 분리: Android(EAS APK), iOS(TestFlight)
- `app version / build number / runtime version` 스크립트 자동 관리

**해결 ② — OTA 업데이트 (EAS Updates)**

- JS 레벨 수정 사항은 스토어 심사 없이 즉시 반영하는 OTA 체계 구축
- 환경별 채널 분리 운영: `preview`(내부 QA/협력사 검증) → `release`(실제 사용자) 단계적 배포
- 적용 사례: 특정 Android 버전 UI 렌더링 오류를 스토어 재배포 없이 즉시 수정 배포

**해결 ③ — Fastlane 스토어 배포 자동화**

- 역할 분리: EAS(빌드 생성 + 스토어 업로드) / Fastlane(심사 제출 + 트랙 관리) 전담
- Android: 내부/공개/프로덕션 트랙 분리 관리, 각 트랙별 심사 제출 자동화
- iOS: TestFlight 이후 App Store 심사 제출 자동화
- Notion MCP를 활용하여 기능 개발 완료 시 관련 API, 변경 사항이 문서에 자동 기록되는 워크플로우 구축

**결과**

- 단순 오타/로직 수정 배포 시간: 수일(심사 기간) → **수 분(OTA)**으로 단축
- 빌드 환경을 코드로 관리 → 배포 시 '환경 차이로 인한 크래시' 원천 차단
- 릴리즈 담당자 개인 의존성 제거 → 배포 안정성과 운영 리스크 동시 감소
- 개발자가 핵심 기능 구현에만 집중할 수 있는 환경 조성
