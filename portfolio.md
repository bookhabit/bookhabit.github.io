### 실시간 채팅 시스템 개발


**포인트**
- 클라이언트와 서버 상태 관리의 일관성 확보
- 낙관적 업데이트 적용으로 사용자에게 즉각적 UI 피드백 제공

**기능 소개**
- STOMP WebSocket 기반 실시간 메시지 송수신 (텍스트, 이미지, 파일, 이모티콘, 멘션 등 9종 이상 메시지 타입 지원)
- FlatList 무한 스크롤로 대용량 메시지 처리
- React Query 캐시와 낙관적 업데이트로 UI 즉시 반영
- 메시지 컴포넌트 구조 확장성 고려한 모듈화 설계

**문제 해결**
- **기존 방식 문제**
  1. 클라이언트 상태와 서버 상태가 분리되어 있어 메시지 전송 후 서버 응답 대기 중 UI 반응 지연
  2. 데이터 무결성 문제
- **개선 방식**
  1. React Query 캐시를 단일 상태 소스로 활용
  2. `setQueryData`를 통한 낙관적 업데이트 적용으로 메시지 전송 즉시 UI 반영

**결과**
- 메시지 로딩 속도 1초 이내로 대폭 개선
- UI 즉각 반영으로 사용자 경험 향상
- 무한 스크롤로 메모리 효율적 대용량 데이터 처리 가능

### 대용량 연락처 기반 초대 시스템


**포인트**
연락처 개수에 비례해 선형적으로 늘어나던 로딩 시간 문제를 백그라운드 동기화 및 로컬 DB 캐싱으로 해결

**기능 소개**
- **기존**: 페이지 진입 시 대용량 연락처 API 호출 후 UI 렌더링 → 긴 로딩시간, 서버 부하, UX 저하
- **개선**: 백그라운드(Headless JS)에서 100개 단위 청크 API 호출하여 로컬 SQLite DB에 저장 → 페이지 진입 시 로컬 DB에서 즉시 UI 렌더링, 서버 호출 최소화

**기술 스택 및 업무**
- React Native, Native Modules (Android/iOS 연락처 접근 및 변경 감지), Headless JS, SQLite, MMKV, Chunked API 요청(100개 단위)
- 연락처 페이지 진입 시 로컬 DB 데이터 확인 후 ContactSyncTask 실행, 앱 포그라운드 복귀 시 동기화 실행, 네이티브 모듈로 연락처 변경 감지 및 동기화 트리거
- 네이티브 권한 체크, 네트워크 상태 확인, 실패 시 롤백 등 예외처리 적용

**문제 해결 과정**
1. 전체 연락처 서버 전송 방식 → 긴 로딩, 높은 서버 부하 문제 발생
2. UI 선노출 + API 점진 호출 방식 → 초기 UX 개선이나 API 요청량, 검색 불일치 문제 미해결
3. 최종 백그라운드 동기화 + 로컬 캐싱 도입 → UX 대폭 개선, 서버 부하 감소, 안정성 확보

**결과**
- 연락처 4,000건 로딩 속도 25초 → 1초 이내로 획기적 개선
- JS 스레드 블로킹 제거로 앱 성능 전반 향상
- 백그라운드 동기화로 앱 진입 시 즉시 UI 제공, 오프라인 환경 지원
- 연락처 변경 감지 → 디바이스 연락처와 서버 DB 동기화 → 서버 API 호출 수 대폭 감소, 트래픽 및 비용 최적화

**얻은 경험**
- 대용량 데이터 백그라운드 처리 및 동기화 전략 실전 적용 경험
- Native 모듈과 React Native 연동, Headless JS 활용 능력 강화
- 사용자 경험 중심 대용량 데이터 처리 설계 및 서버 부하 최적화 경험

### Offline 지원 및 성능개선을 위한 SQLite 도입


**포인트**
React Native 앱에 SQLite 기반 로컬 DB를 도입하여 서버 API 의존도를 줄이고, 오프라인 환경에서도 빠르고 안정적인 UX를 제공하는 로컬 우선 구조 구현

**기능 소개**
- **기존 방식**: 화면 진입 시마다 서버 API 호출 → 로딩 지연 및 불필요한 네트워크 요청 다수 발생
- **개선 방식**: 로컬 DB에 데이터 저장 후, UI는 로컬 DB 우선 조회 → 서버 호출은 비동기 처리로 UX 지연 최소화

**결과**
- 오프라인 상태에서도 주요 기능 정상 동작 보장
- 서버 API 호출 횟수 및 트래픽 대폭 감소
- 실시간 데이터 변경 시 로컬 DB 및 React Query 캐시 동기화로 UI 일관성 확보

### 알림 기능


**알림 시스템 구조 설계 및 개선**

**포인트**
- FCM 리소스 낭비를 줄이기 위해 **분산 알림 구조** 설계
  - 포그라운드에서 소켓으로 받은 메시지는 로컬 알림(notifee)으로 처리
  - 백그라운드 알림은 FCM으로 처리
- 메시지 외 알림 타입이 다양해져 복잡해진 구조를 **TypeScript 타입 정의**로 엄격히 관리하여 컴파일 단계에서 오류를 사전에 차단, 유지보수성 향상

**문제 및 해결방안**
1. **문제**
   - 모든 알림을 FCM으로 통일해 처리 → 포그라운드에서도 불필요하게 FCM 경유
   - 리소스 낭비 및 반응 지연 발생
   - 알림 설정 반영 로직이 단순해 유지보수는 쉽지만 성능 효율이 떨어짐

2. **해결방안**
   - **분산 처리 방식** 채택
     - 포그라운드: WebSocket으로 메시지 수신 후 로컬 알림(notifee) 즉시 표시
     - 백그라운드: FCM 푸시 알림 수신 후 표시
   - 알림 소스·타입별로 구조화된 TypeScript 인터페이스 설계
     - `NotificationClickEvent`, `NotificationListType` 등으로 알림 데이터 형식 표준화 및 확장성 고려

**결과**
- 포그라운드 알림 속도 개선 및 FCM 리소스 절감
- 다양한 알림 타입에도 일관된 구조 유지 가능
- TypeScript 기반의 엄격한 타입 관리로 유지보수 부담 감소 및 확장성 확보

### 인증/인가 E2E 테스트


#### 1. 테스트 코드를 작성한 실무적 이유

##### ① QA 리소스를 "사람"이 아닌 "시스템"으로 대체

인증(회원가입 / 로그인)은
- 모든 유저가 반드시 거치는 핵심 플로우
- 한 번 깨지면 서비스 신뢰도에 직접적인 타격이 발생

하지만 실무에서는
- QA 인력이 충분하지 않거나
- 개발자가 직접 수동 테스트를 반복하는 경우가 많음

이로 인해 매 배포마다:
- 동일한 시나리오를 직접 입력
- 버튼 클릭
- 에러 케이스 재확인

하는 **비생산적인 작업**이 반복됨

→ **E2E 테스트로 인증 플로우를 자동화하여, QA 시간을 코드 실행 시간으로 치환**

##### ② "리팩터링이 두려운 코드"를 "변경 가능한 코드"로 만들기

인증 로직은:
- UI 변경
- API 스펙 변경
- 에러 처리 로직 추가

등의 수정이 빈번함

하지만 테스트가 없을 경우:
- 작은 UI 수정에도
- "로그인이 혹시 깨지지 않았을까?"라는 불안으로

수동 검증이 필수

→ E2E 테스트를 통해 **리팩터링 이후에도 핵심 기능이 정상 동작함을 즉시 검증 가능**

##### ③ Form / API 에러 처리를 명세 수준으로 고정

실무에서 자주 발생하는 문제:
- Form validation 규칙이 코드마다 다르게 적용됨
- API 에러 메시지 처리 방식이 화면마다 달라짐

E2E 테스트에서:
- 필수 입력값 누락
- 잘못된 형식 입력
- API 실패 응답

을 **시나리오로 고정**

→ "의도한 UX"가 테스트 코드로 명확히 문서화됨

#### 2. E2E 테스트에서 실제로 겪은 문제와 대응 (실무 디테일)

##### 문제 ① 플랫폼별 UI 동작 차이로 테스트 실패

- iOS / Android 간
  - 키보드 노출 방식
  - 입력 포커스 이동
  - 스크롤 위치

차이로 동일한 테스트 스크립트가 실패

**대응**
- 입력 후 스크롤 이동 로직 명시
- UI 상태를 기준으로 다음 액션 수행하도록 수정

→ 테스트가 "기기 의존적"이 아닌 "상태 기반"으로 동작

##### 문제 ② API 응답 지연 + 애니메이션 타이밍으로 flaky 테스트 발생

- 네트워크 상태에 따라
  - API 응답 타이밍 변동
  - 애니메이션 완료 전 assertion 실행

**대응**
- 고정 delay 제거
- API 응답 안정화 시점 기준으로 테스트 진행
- 이메일 인증 플로우는 "발송 성공 → 인증 완료 상태 가정"으로 분리 테스트

→ flaky 테스트 제거, CI 환경에서도 안정적으로 통과

#### 3. 실질적인 성과 (가장 중요한 부분)

##### ① 배포 전 인증 기능 회귀 여부를 **수 분 내 확인 가능**

- 리팩터링
- UI 수정
- API 변경

이 발생해도

→ **테스트 실행만으로 로그인/회원가입 정상 여부 즉시 판단**

##### ② API 변경 시 프론트/서버 커뮤니케이션 비용 감소

- 테스트 실패 지점이 곧
  - UI 문제인지
  - API 응답 문제인지

를 명확히 알려줌

→ "이거 서버 문제인가요?" 같은 불필요한 커뮤니케이션 감소

##### ③ QA 인력이 없어도 신뢰 가능한 검증 체계 확보

- 사람이 직접 테스트하지 않아도
- 핵심 유저 플로우에 대해
  - 일관된 품질 보장 가능

→ 소규모 팀 / 1인 개발 환경에서도 **엔터프라이즈 수준의 안정성 확보**

#### 4. 한 줄 요약 (포트폴리오 / 면접용)

**포트폴리오용**
> 인증 기능을 E2E 테스트로 자동화하여, QA 리소스 없이도 핵심 유저 플로우의 안정성을 지속적으로 검증할 수 있는 구조를 구축했다.

**면접 구두 설명용**
> 테스트 코드를 작성한 이유는 '버그를 찾기 위해서'가 아니라, **리팩터링과 배포를 두려움 없이 할 수 있는 환경을 만들기 위해서**였습니다.

## 복지찬스

### 네이티브(Kotlin, Swift)로 만보기 기능 자체 개발

**서비스 개요**
- React Native 기반 만보기 서비스
- Android Foreground Service를 활용한 백그라운드 걸음 수 측정
- 네이티브(Android) + React Native 하이브리드 구조
- 실제 사용자 대상 운영 경험 보유 (3개월)

#### 주요 문제점 및 기술적 이슈

##### Android Foreground Service 관련 비정상 종료

**문제**
- Android 12+ 환경에서 Foreground Service 실행 중 비정상 종료 발생
- `ForegroundServiceStartNotAllowedException` 등 시스템 제약 이슈
- 네이티브 코드 수정 시 앱 크래시 및 ANR 빈번 발생

**원인**
- Android OS 버전별 Foreground Service 정책 차이
- 앱 상태(background / killed)에 따른 서비스 실행 타이밍 문제
- Service, Module, Database 인스턴스 관리 미흡

##### 걸음 수 동기화 문제

**문제**
- 알림 센터에 표시되는 걸음 수와 React Native 화면(UI)의 걸음 수가 불일치
- 앱 재진입, 백그라운드 복귀 시 상태 꼬임 발생

**구조적 원인**
- **네이티브(Android)**
  - Coroutine 기반 비동기 처리
  - Service / Native Module / Database 간 상태 관리 복잡
  - Singleton 사용으로 인한 생명주기 관리 어려움
- **React Native**
  - 네이티브에서 변경된 값이 즉시 UI에 반영되지 않음
  - 앱 상태(AppState)에 따른 동기화 누락

#### 문제 해결 및 개선 사항

##### Foreground Service 안정화

- Android OS 버전별 서비스 실행 정책 분기 처리
- Service 시작 시점 명확화 (앱 상태 기반 제어)
- Service / Module / DB 접근 구조 재정비
- 네이티브 코드 수정 시 크래시 로그 기반 원인 추적 및 재현 환경 구성

**결과**
- Foreground Service 관련 크래시 및 ANR 발생률 유의미하게 감소
- 장시간 백그라운드 동작 안정성 확보

##### 걸음 수 동기화 구조 개선

- 네이티브 DB를 **Single Source of Truth**로 설정
- Service → DB → React Native 흐름으로 데이터 단방향화
- 앱 포그라운드 진입 시 DB 기준으로 UI 상태 재동기화
- 이벤트 기반 네이티브 → JS 브리지 통신 정리

**결과**
- 알림 센터 / 앱 UI 간 걸음 수 불일치 문제 해결
- 앱 재실행, 백그라운드 복귀 시에도 일관된 상태 유지

### 비정상 발생 ANR 대응 (Firebase Crashlytics)

**문제 정의**
- Android 12+ 환경에서 **비정상 종료 및 ANR 지속 발생**
- Foreground Service 실행 중 시스템 제약으로 앱 종료
- 네이티브 코드 수정 이후 크래시 발생 빈도 증가

**대표 오류**
- `ForegroundServiceStartNotAllowedException`

**원인 분석**
- Android OS 버전별 Foreground Service 정책 차이
- 앱 상태(background / killed)에 따른 서비스 실행 제한
- Service 시작 타이밍 및 생명주기 관리 미흡
- 네이티브 코드 변경 시 영향 범위 파악 어려움

**대응 전략**

##### Foreground Service 안정화
- Android OS 버전별 서비스 실행 정책 분기 처리
- 앱 상태 기반 서비스 시작 시점 명확화
- Foreground Service 실행 흐름 재정비
- 크래시 발생 지점 기준으로 코드 영향 범위 축소

##### ANR 모니터링 환경 구축
- Android Vitals 기본 지표 활용
- Firebase Crashlytics 연동
  - 크래시
  - ANR
  - 네이티브 스택 트레이스 수집

**대응 프로세스**
1. Crashlytics를 통해 ANR / 크래시 탐지
2. 스택 트레이스 분석으로 문제 코드 식별
3. Android Studio Logcat / Profiler로 재현
4. 단계적 수정 후 재배포 및 재모니터링

**결과**
- Foreground Service 관련 크래시 및 ANR 발생률 감소
- Android 12+ 환경에서 서비스 안정성 확보
- 운영 중 장애를 모니터링 기반으로 빠르게 대응 가능한 구조 구축

### 프로덕션·협력사 환경 분리를 위한 모바일 빌드/버전 관리 시스템 구축

**구축 배경**
- 프로덕션 환경과 외부 협력사 테스트 환경을 **동시에 개발·검증해야 하는 상황**
- 기존 로컬 빌드 기반 배포 방식으로 인해 다음과 같은 문제가 발생
  - 개발자별 로컬 환경 차이로 인한 동작 불일치
  - 날짜 기반 파일명 관리로 인한 **버전 혼선** (동일 날짜 내에서도 서로 다른 빌드 존재)
  - QA 및 외부 협력사와의 커뮤니케이션 비용 증가

**해결 ① – EAS 클라우드 빌드 환경 구축**
- Expo **EAS Build** 기반 클라우드 빌드 환경 도입
- 팀원 및 외부 협력사와 **동일한 빌드 결과를 일관되게 공유**
- 플랫폼별 빌드 프로파일 분리 운영
  - **Android**: EAS APK 배포
  - **iOS**: TestFlight 비공개 테스트 활용

**효과**
- 개발자별 로컬 환경 차이 제거
- 빌드 및 테스트 공유 속도 개선
- 외부 협력사와의 QA 커뮤니케이션 비용 감소

**해결 ② – 플랫폼별 버전 관리 자동화**
- 동일 코드베이스에서 **환경별 버전 정책 분리 관리**
  - Android: `versionCode`
  - iOS: `buildNumber`
- `app version / build number / runtime version`을 스크립트로 자동 관리
- 프로덕션 / 협력사 테스트 / 내부 검증 버전을 명확히 구분하여 운영

**효과**
- 버전 충돌 및 휴먼 에러 방지
- 스토어 심사 및 릴리즈 과정의 안정성 확보
- 다중 환경 운영 시 배포 신뢰도 향상

**정리**
- 클라우드 빌드 도입을 통해 **환경 차이에 의한 오류를 구조적으로 제거**
- 버전 관리 자동화를 통해 **다중 환경·다중 이해관계자 협업에 적합한 배포 체계 구축**
- 실제 운영 환경에서 **협력사 테스트, QA, 프로덕션 릴리즈를 안정적으로 병행**

### OTA 업데이트 (EAS Updates)를 활용한 긴급 버그 대응

**적용 배경**
- 스토어 심사 없이 **JS 레벨 수정 사항을 즉시 사용자에게 배포할 필요**
- 운영 중 발견되는 경미하지만 긴급한 이슈에 대해 스토어 심사 대기(수시간~수일) 없이 대응 요구
- 잦은 스토어 재배포로 인한 릴리즈 피로도 증가 및 사용자 불편 발생

**적용 방식**

##### OTA 적용 범위 명확화
- **JS 코드 수정으로 해결 가능한 이슈만 OTA 대상으로 한정**
  - UI 깨짐
  - 잘못된 분기 로직
  - 특정 조건에서만 발생하는 오류 처리
- 네이티브 변경이 필요한 경우는 기존 스토어 배포 프로세스 유지

##### 환경별 업데이트 채널 분리 운영
- EAS Updates 채널을 **preview / release 환경으로 분리**
  - `preview`: 내부 QA 및 테스트용, 협력사 검증 및 사전 확인 목적
  - `release`: 실제 사용자 대상 배포
- preview 채널에서 문제 없는 경우에만 release 채널로 반영

##### 스토어 배포와 OTA 역할 분리
- **스토어 배포**: 네이티브 변경, 기능 추가, 구조 변경
- **OTA 배포**: 긴급 버그 수정, 운영 중 발견된 경미한 이슈 대응

**적용 사례**
- 특정 Android 버전에서만 발생하던 **UI 렌더링 오류**
  - JS 조건 분기 수정으로 해결 가능
  - 스토어 재배포 없이 OTA로 즉시 수정 사항 배포
- 운영 중 사용자 피드백으로 확인된 잘못된 상태 처리 로직
  - 앱 재실행 없이 OTA로 수정 반영

**결과**
- JS 수정으로 해결 가능한 이슈를 **즉시 사용자에게 반영**
- 긴급 버그 대응 시 스토어 심사 의존도 감소
- preview → release 단계적 OTA 배포로 운영 리스크 최소화 및 배포 안정성 확보
- 운영 중 사용자 불편에 대한 대응 속도 및 신뢰도 향상

### 개발·배포·운영 전반의 프로세스 자동화 및 표준화

> 배포 자동화가 릴리즈 단계의 표준화라면, Notion MCP 기반 워크플로우는 개발 과정 자체를 표준화하는 것을 목표로 했다.

> 배포(Fastlane)와 개발 문서(Notion MCP)를 각각 자동화한 것이 아니라, **개발 → 문서 → 릴리즈까지 사람이 개입하지 않아도 일관되게 동작하는 운영 체계를 구축**했다.

#### 핵심 구현 ① — Fastlane 활용한 Android / iOS 스토어 배포 자동화

**적용 배경**
- Android / iOS 앱 배포 과정에서
  - 스토어 심사 제출
  - 트랙 선택 (내부 / 공개 / 프로덕션)
  - 릴리즈 상태 관리
  가 매 릴리즈마다 반복적으로 발생
- 빌드 및 업로드는 **EAS(Build / Submit)** 로 처리하고 있었으나, 스토어 심사 단계는 여전히 수동 작업에 의존
- 릴리즈 담당자의 숙련도에 따라 실수 가능성 존재

**구현 방식**
- **역할 분리 기반 배포 자동화 설계**
  - **EAS**: Android / iOS 빌드 생성, 스토어 업로드 (binary 전달)
  - **Fastlane**: 스토어 심사 제출 및 트랙 관리 전담
- Fastlane을 활용하여 다음 작업을 자동화
  - **Android**: 내부 테스트 / 공개 테스트 / 프로덕션 트랙 분리 관리, 각 트랙별 심사 제출 자동화
  - **iOS**: TestFlight 이후 App Store 심사 제출 자동화
- 플랫폼별로 lane을 분리하여 Android / iOS 배포 흐름을 독립적으로 관리, 릴리즈 단계에서 실행 명령만으로 심사 제출 가능하도록 구성

**핵심 성과**
- 스토어 심사 제출 과정을 **단일 명령 기반으로 표준화**
- 심사 단계에서 발생하던 트랙 선택 오류, 수동 설정 실수를 구조적으로 차단
- 릴리즈 담당자 개인에게 의존하던 배포 구조를 제거하여 **배포 안정성과 운영 리스크 감소**를 동시에 확보

> Fastlane을 도입하여 Android/iOS 스토어 심사 제출을 스크립트로 표준화함으로써, 릴리즈 담당자 의존성을 제거하고 배포 안정성을 확보하였다.

#### 핵심 구현 ② — Notion MCP 기반 문서 중심 개발 및 협업 규칙 표준화

**문제 인식**
- 초기 개발 단계에서
  - API 명세 동기화 체계 부재
  - 서버–프론트 간 API 규칙 미정의
- 현재는 1인 개발 환경이지만, 팀 확장 시
  - 문서 불일치
  - API 사용 방식 혼선
  - 코드 스타일 파편화
  가 발생할 가능성 높다고 판단

**접근 전략**
- **문서를 단순 참고 자료가 아닌 '시스템의 기준'으로 격상**
- API 명세를 **Single Source of Truth**로 정의
- Notion MCP를 활용하여 문서, 코드 규칙, 개발 결과 기록을 하나의 흐름으로 연결

**구현 내용**
- API 명세 작성 규칙 및 MD 파일 구조 정의
- 명세를 기준으로 API Hook 패턴, 요청/응답 타입 구조를 통일
- MCP를 통해 동일한 규칙의 코드 스타일로 서버와 협업 가능하도록 설계
  - 프론트엔드 코드 가독성 및 유지보수성 향상
- 기능 개발 완료 시 관련 API, 변경 사항, 개발 결과가 **Notion 문서에 자동 기록되는 워크플로우 구축**

**핵심 성과**
- 문서와 코드 간 불일치 문제를 구조적으로 예방
- 개발 완료 후 문서 정리에 소요되던 시간 대폭 감소
- 향후 신규 팀원이 합류하더라도 API 사용 규칙, 코드 스타일, 문서 구조를 빠르게 이해할 수 있는 **협업 시스템 기반 마련**
